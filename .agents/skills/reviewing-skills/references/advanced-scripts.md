## 解決する・逃げ出さない

スクリプトはAIに任せるのではなくエラー条件を処理する。

**OK（エラーを明示的に処理）：**
```js
const fs = require('fs');

function processFile(path) {
  try {
    return fs.readFileSync(path, 'utf8');
  } catch (err) {
    if (err.code === 'ENOENT') {
      console.log(`ファイル${path}が見つかりません、デフォルトを作成しています`);
      fs.writeFileSync(path, '');
      return '';
    } else if (err.code === 'EACCES') {
      console.log(`${path}にアクセスできません、デフォルトを使用しています`);
      return '';
    }
    throw err;
  }
}
```

**NG（AIに任せる）：**
```js
function processFile(path) {
  return require('fs').readFileSync(path, 'utf8'); // 失敗してAIに理解させる
}
```

**設定パラメータも正当化・文書化する：**

OK（自己文書化）：
```js
// HTTPリクエストは通常30秒以内に完了する
// より長いタイムアウトは遅い接続を考慮
const REQUEST_TIMEOUT = 30;

// 3回の再試行は信頼性と速度のバランスを取る
// ほとんどの一時的な失敗は2回目の再試行で解決
const MAX_RETRIES = 3;
```

NG（マジックナンバー）：
```js
const TIMEOUT = 47; // なぜ47？
const RETRIES = 5;     // なぜ5？
```

## ユーティリティスクリプトを提供する

**推奨ランタイム：** スキルのスクリプトは **Node.js（JavaScript）** または **bash** で書くことを推奨する。これらは利用可能なランタイムとして広く揃っているため。Pythonは環境によって利用できない場合がある。

AIがスクリプトを書けたとしても、事前作成されたスクリプトには利点がある：
- 生成されたコードより信頼性が高い
- トークンを節約する（コンテキストにコードを含める必要がない）
- 使用全体で一貫性を確保する

**実行意図を明確にする：**
- **実行する**（最も一般的）：「analyze_form.pyを実行してフィールドを抽出する」
- **参照として読む**（複雑なロジックの場合）：「フィールド抽出アルゴリズムについてはanalyze_form.pyを参照してください」

**スクリプトのドキュメント例：**
```markdown
## ユーティリティスクリプト

**analyze_form.js**：PDFからすべてのフォームフィールドを抽出

```bash
node scripts/analyze_form.js input.pdf > fields.json
```

出力形式：
```json
{
  "field_name": {"type": "text", "x": 100, "y": 200},
  "signature": {"type": "sig", "x": 150, "y": 500}
}
```

**validate_boxes.js**：重複する境界ボックスをチェック

```bash
node scripts/validate_boxes.js fields.json
# 戻り値：「OK」または競合をリスト
```
```

## ビジュアル分析を使用する

入力を画像としてレンダリングできる場合、AIに分析させる：

```markdown
## フォームレイアウト分析

1. PDFを画像に変換する：
   ```bash
   node scripts/pdf_to_images.js form.pdf
   ```

2. 各ページ画像を分析してフォームフィールドを特定する
3. AIはフィールドの場所とタイプを視覚的に確認できる
```

## 検証可能な中間出力を作成する

**「計画-検証-実行」パターン**：AIが先に構造化された計画を作成し、実行前にスクリプトで検証することで早期にエラーをキャッチする。

バッチ操作・破壊的変更・複雑な検証ルール・高リスク操作に特に有効。

**ワークフロー例：**
```
分析 → 計画ファイルを作成 → 計画を検証 → 実行 → 確認
```

**このパターンが機能する理由：**
- 早期にエラーをキャッチする（変更を適用する前に問題を発見）
- 機械検証可能（スクリプトが客観的な検証を提供）
- 可逆的な計画（元のファイルに触れずに計画を反復できる）
- 明確なデバッグ（エラーメッセージが特定の問題を指す）

**検証スクリプトのヒント：** 「フィールド'signature_date'が見つかりません。利用可能なフィールド：customer_name、order_total」のような具体的なエラーメッセージにするとAIが問題を修正しやすい。

## パッケージ依存関係

プラットフォーム別の制限：
- **claude.ai**：npmとPyPIからパッケージをインストールできる、GitHubリポジトリからプルできる
- **Anthropic API**：ネットワークアクセスなし、ランタイムパッケージインストールなし

SKILL.mdで必要なパッケージをリストし、利用可能であることを確認する。インストール方法を明示する。

## ランタイム環境とファイルアクセス

**AIがスキルにアクセスする方法：**
1. 起動時にYAMLフロントマターからname・descriptionがロードされる
2. 必要に応じてbash読み取りツールでSKILL.mdや他ファイルにアクセスする
3. スクリプトはコンテンツをコンテキストにロードせずにbashで実行できる（スクリプトの出力のみトークンを消費）
4. 参照ファイルは実際に読まれるまでコンテキストトークンを消費しない

**ファイル設計の指針：**
- ファイルパスにフォワードスラッシュを使用（`reference/guide.md`）
- コンテンツを示す説明的な名前（`form_validation_rules.md` > `doc2.md`）
- ドメインまたは機能別にディレクトリを構造化
  - OK：`reference/finance.md`、`reference/sales.md`
  - NG：`docs/file1.md`、`docs/file2.md`

## MCPツール参照

MCP（モデルコンテキストプロトコル）ツールを使用する場合、「ツールが見つかりません」エラーを避けるために完全修飾ツール名を使用する。

**形式：`ServerName:tool_name`**

```markdown
BigQuery:bigquery_schemaツールを使用してテーブルスキーマを取得する。
GitHub:create_issueツールを使用して問題を作成する。
```

サーバープレフィックスがないと、特に複数のMCPサーバーが利用可能な場合、AIはツールの場所を特定できない可能性がある。
