# Skill Reviewer

指定されたスキル（SKILL.mdとそのディレクトリ構造）をベストプラクティスに基づいて評価し、具体的な改善提案を提供する。

## 評価の進め方

以下の6つのステップを順番に実行する。各ステップで問題を発見したら即座に記録しておき、最後に総合評価としてまとめる。

```
レビュー進捗：
- [ ] ステップ1：YAMLフロントマターの確認
- [ ] ステップ2：SKILL.mdの構造と簡潔さの確認
- [ ] ステップ3：コンテンツの質の確認
- [ ] ステップ4：ワークフローとフィードバックループの確認
- [ ] ステップ5：コードとスクリプトの確認（該当する場合）
- [ ] ステップ6：総合評価とチェックリスト
```

---

## ステップ1：YAMLフロントマターの確認

`name`、`description`、`argument-hint` フィールドを検証する。詳細ルールは [references/skill-structure.md](references/skill-structure.md) の「命名規則」「効果的な説明の書き方」「argument-hintの書き方」を参照。

**確認項目：**
- `name`：64文字以内、小文字・数字・ハイフンのみ、予約語（anthropic・claude）を含まない
- `name`：**動名詞形（例：`processing-pdfs`）が必須。** 名詞句・アクション形は不可
- `name`：曖昧な名前（`helper`、`utils`、`tools`）や過度に一般的な名前（`data`、`files`）でない
- `name`：**意味的正確さ**：スペルミスや語義の混同がないか確認する。文法形式が正しくても、意図した意味を正確に表しているかをチェックする
  - 例：`turning-prompts`（turning=回転）は文法上は動名詞形だが、「tuning=調整」との混同の可能性がある
  - **重要度の判断基準：文法形式の違反（動名詞形でない）は重大な問題。意味的不正確さ（スペルミスや語義の混同の疑い）は軽微な問題として分類し、「意図した動詞が何かを推測した上で提案する」形で指摘する**
- `description`：空でない、1024文字以内
- `description`：三人称（スキルを主語とした客観的な記述）で書かれている。AIや自分自身を主語にした一人称的・能力表明的な表現はNG
  - OK：「〜を処理してレポートを生成します」（スキルの動作を客観的に記述）
  - NG：「〜をお手伝いできます」「〜することができます」（能力・意志の表明）
- `description`：スキルが何をするか＋いつ使用するかの両方が含まれている
- `description`：具体的なキーワードが含まれている（曖昧な「データを処理します」はNG）
- `argument-hint`：引数がある場合、**必ずクォートで囲む**（例：`"[arg1] [arg2]"`）
- `argument-hint`：クォートなし（`[arg1] [arg2]`）は一部のAIで解釈エラーが起きるため不可

## ステップ2：SKILL.mdの構造と簡潔さの確認

詳細は [references/core-principles.md](references/core-principles.md) と [references/skill-structure.md](references/skill-structure.md) の「段階的開示パターン」を参照。

**確認項目：**
- SKILL.mdボディが500行以下か
- エージェントがすでに知っていることを不必要に説明していないか（冗長な説明がないか）
- 追加の詳細情報は別ファイルに分離されているか
- 参照ファイルへのリンクがSKILL.mdから1レベル深いか（深いネストはNG）
- 100行以上の参照ファイルには目次があるか
- ディレクトリ構造が発見しやすい名前になっているか（`file1.md`はNG、`form_validation_rules.md`はOK）

---

## ステップ3：コンテンツの質の確認

詳細は [references/content-guidelines.md](references/content-guidelines.md) を参照。

**確認項目：**
- 時間に敏感な情報（「2025年8月以降は...」など）が含まれていないか
- 含まれている場合、「古いパターン」セクションに隔離されているか
- スキル全体で一貫した用語を使用しているか（「APIエンドポイント」と「URL」を混用していないか）
- 例が抽象的でなく具体的か
- Windowsスタイルのパス（`\`）が使われていないか（フォワードスラッシュのみOK）
- 多くのオプションを提示しすぎていないか（デフォルトを1つ示し、例外を追記するスタイル推奨）

---

## ステップ4：ワークフローとフィードバックループの確認

詳細は [references/patterns.md](references/patterns.md) を参照。

**確認項目：**
- 複雑なタスクが明確なステップに分割されているか
- チェックリスト形式（AIが進捗を追跡できる）が活用されているか
- バリデーション→修正→繰り返しのフィードバックループが実装されているか
- 条件分岐がある場合、分かりやすく示されているか
- テンプレートや例が出力品質の向上に活用されているか

---

## ステップ5：コードとスクリプトの確認（該当する場合）

コード・スクリプトが含まれていない場合はスキップ。詳細は [references/advanced-scripts.md](references/advanced-scripts.md) を参照。

**確認項目：**
- スクリプトがエラーを処理しているか（AIに任せていないか）
- マジックナンバーが説明付きか（`TIMEOUT = 47` はNG、理由付きであればOK）
- 必要なパッケージが明示されているか（インストール方法も含む）
- スクリプトの実行意図が明確か（「実行する」vs「参照として読む」）
- MCP ツールを使う場合、`ServerName:tool_name` 形式で完全修飾されているか
- 重要な操作に検証ステップがあるか

---

## ステップ6：総合評価とチェックリスト

[references/checklist.md](references/checklist.md) の完全なチェックリストを使用して総合評価を行う。

**評価レポートの形式：**

```
## レビュー結果：[スキル名]

### 総合評価
[優良 / 良好 / 要改善 / 要大幅改善]

### 良い点
- （具体的に列挙）

### 問題点と改善提案
#### 重大な問題
- （修正が必要なもの）

#### 軽微な問題
- （改善推奨のもの）

### チェックリスト結果
（checklist.mdのカテゴリ別合否）

### 次のアクション
1. （優先度順に具体的なアクション）
```
