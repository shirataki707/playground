# Prompt Tuning（パレート探索型）

**指示文の品質は書いた本人には分からない。** そして「1つのベスト候補を上書きし続けるGreedy探索」は局所最適解に陥りやすく、有望な候補を早期に捨てる。

このスキルは3つの仕組みで局所最適を回避する:

| 仕組み | 内容 |
|---|---|
| **候補プール（Pareto）** | 複数の候補を並行維持し、互いに支配しない集合（パレートフロンティア）を保つ |
| **Reflection（変異）** | LLM自身に実行結果を分析させ、次の候補を生成させる |
| **Merge（統合）** | 別シナリオに特化した2候補を統合した新候補を生成する |

## いつ使うか

**使う場面:**
- エージェントへの指示ファイル（スキルの SKILL.md や、ルールファイル）を新規作成・大幅改訂した直後
- Greedy探索で改善が頭打ちになったとき
- 頻繁に使う重要な指示を堅牢にしたいとき

**使わない場面:**
- 一回限りの使い捨てプロンプト（評価コストが割に合わない）
- 好みを反映したいだけで品質改善が目的でないとき

## コア概念

### パレート支配とフロンティア

候補 X が候補 Y を「支配する」条件: X がすべての評価軸で Y 以上かつ、少なくとも1軸で厳密に優れている。

**パレートフロンティア** = 互いに支配しない候補の集合。

```
例：
- C-001: シナリオA 精度95%、シナリオB 精度60%
- C-002: シナリオA 精度70%、シナリオB 精度90%
→ どちらも相手を支配しない → 両方をフロンティアとして維持
```

支配された候補はプールから除外する。Greedy探索と違い、フロンティアは縮小しないかぎり複数候補を保ち続ける。

### Reflection（LLMによる変異）

実行レポートを受け取った専用の Reflection Agent が次の候補案を生成する。人間が手動で修正点を考えるのではなく、LLMに分析・提案させる。

### Merge（統合）

フロンティア上に「シナリオAに強いX」と「シナリオBに強いY」が並存するとき、両方の強みを統合した C-merge を生成する。

## ワークフロー

**最初に必ずユーザーに確認する。** このスキルは複数の Subagent を並列・反復実行するため、トークンを大量消費する。ユーザーが意図的に起動した場合でも、実行前に以下を提示して明示的な承認を得る:

> このスキルは Subagent を複数回並列実行するため、トークンを大量消費します。続行しますか？

ユーザーが「はい」と答えた場合のみ Step 0 に進む。「いいえ」または応答がなければ終了する。

```
進捗チェックリスト:
- [ ] ユーザー確認（トークン消費の警告と承認）
- [ ] Step 0: description と body の整合チェック（静的）
- [ ] Step 1: ベースラインの準備（シナリオ・要件・初期候補 C-000）
- [ ] Step 2: 各候補を Subagent で並列実行
- [ ] Step 3: 評価 + パレートフロンティア更新
- [ ] Step 4: Reflection → 新候補を生成（変異）
- [ ] Step 5: Merge（フロンティアに2候補以上あるとき）
- [ ] Step 6: 繰り返す
- [ ] Step 7: 収束判定
```

### Step 0: description と body の整合チェック

Subagent を起動する前に確認する。frontmatter の `description` が謳う内容と body の内容が一致しているか。乖離があれば Step 1 に進む前に修正する。

例（乖離あり）: description に「Android のレイアウト修正を行います」とあるのに body が Kotlin のデバッグ手順だけ

### Step 1: ベースラインの準備

**評価シナリオ（2〜3つ）:**
- 中央値シナリオ（実際によくある使用場面）× 1
- エッジケース（やや難しい / 珍しい状況）× 1〜2

Pococha Android の例:
- 中央値: 「新規画面に ViewModel を追加する」
- エッジ: 「既存の ViewModel が肥大化しているのでリファクタリングする」

**要件チェックリスト（シナリオごとに3〜7項目）:**
- `[critical]` タグを最低1つ付ける（全部○のときのみ「成功」と判定する）
- 事前に確定させ後から変えない

**初期候補 C-000**: 現在の指示文をそのまま C-000 として候補プールに登録する。

### Step 2: 各候補を Subagent で並列実行

**必ず新規 subagent を dispatch する**（自分で読み直すのはNG）。

- 各候補 × 各シナリオの組合せを**1つのメッセージで並列実行**する
- 例: 候補2つ × シナリオ2つ = Agent 呼び出し4本を同時送信
- Subagent プロンプトは後述の「Subagent 実行テンプレート」を使う

### Step 3: 評価 + パレートフロンティア更新

結果を「候補別評価表」に記録し（フォーマットは後述）、パレートフロンティアを更新する。

**評価軸:**

| メトリクス | 取り方 | 意味 |
|---|---|---|
| 成功/失敗 | `[critical]` 項目が全部○なら成功 | 最低ライン |
| 精度 | 要件を何%満たしたか（部分的=0.5点） | 部分成功の程度 |
| ステップ数 | Agent tool の `tool_uses` | 指示の効率 |
| 所要時間 | Agent tool の `duration_ms` | 認知負荷の代替指標 |
| 再試行回数 | Subagent の自己申告から抽出 | 指示の曖昧さのシグナル |

**重み付け:** 質的フィードバック（詰まった箇所・補完箇所）を主、数値は補助とする。数値だけ追うと指示が削られすぎる。

**ステップ数のシグナル読み方:** シナリオ間で他の3〜5倍以上なら、そのシナリオ用の手順が指示文に欠けているサイン。精度100%でもこの偏りがあれば次のイテレーションに進む。

**フロンティア更新手順:**
1. 全候補ペアで支配チェック（Ci が Cj を支配するか）
2. 支配された候補をプールから除外
3. 残った集合が新しいパレートフロンティア

### Step 4: Reflection → 新候補生成（変異）

フロンティア上の各候補について、専用の **Reflection Agent** を dispatch する（フロンティア候補1つにつき1 Agent）。

Reflection Agent の役割:
- 実行レポート（詰まった箇所・補完箇所・メトリクス）を読む
- その候補の最大の弱点を1つ特定する
- 弱点を修正した新候補（変異体）の指示文全文を生成する

「1イテレーション1テーマ」を Reflection Agent 内でも守る。テンプレートは後述。

### Step 5: Merge（フロンティアに2候補以上あるとき）

次の条件が揃ったとき Merge を実行する:
- フロンティアに2候補以上存在する
- 各候補が**異なるシナリオに特化**している（C-X がシナリオAに強く、C-Y がシナリオBに強い）
- 両候補が少なくとも2イテレーション以上生き残っている（成熟した候補同士を統合する）

Merge の手順:
1. C-X と C-Y の指示文全文を並べて専用の **Merge Agent** に渡す
2. 「C-X の強みをここ、C-Y の強みをここ」と統合した C-merge の指示文全文を生成させる
3. C-merge を候補プールに追加し、Step 2〜3 で評価する

C-merge が C-X と C-Y の両方を支配すれば Merge 成功。そうでなければ C-merge もフロンティアに加えてそのまま並行維持する。

### Step 6: 繰り返す

**別の新規 subagent**（前回とは別）で Step 2〜5 を繰り返す。前回の subagent を再利用しない（学習バイアスが入るため）。

### Step 7: 収束判定

次の条件を**連続2回**満たしたら停止する:
- フロンティアに新候補が追加されない（変異も Merge も改善なし）
- 新しく出てきた不明瞭点: 0件
- 全候補の精度の前回比改善: +3ポイント以下

停止前に未使用シナリオで1回テスト（過適合チェック）。精度が直近平均から15ポイント以上落ちたら過適合 → ベースラインのシナリオ設計に戻ってエッジを追加する。

**停止できないとき:** 3回以上フロンティアが縮小しない場合は設計を書き直す。

## Subagent 実行テンプレート

```
あなたは [候補ID: C-XXX] の指示文を初めて読む実行者です。前提知識はありません。

## 対象の指示文（C-XXX）
[指示文の全文を貼る、またはパスを指定して Read させる]

## シナリオ
[具体的な状況設定を 1 段落で書く]

## 要件チェックリスト
1. [critical] [最重要項目]
2. [通常項目]
3. [通常項目]
...
（[critical] は最低 1 つ必須。判定は ○ / × / 部分的 の 3 値。）

## タスク
1. 上記の指示文に従ってシナリオを実行し、成果物を作る
2. 終了後、以下のレポートを返す

## レポート形式
- 成果物: [生成物または実行結果のサマリ]
- 要件達成: 各項目について ○ / × / 部分的（理由付き）
- 詰まった箇所: 指示のどこが分かりにくかったか（箇条書き。なければ「なし」）
- 補完した箇所: 指示に書いていなかったが自分の判断で決めた部分（箇条書き）
- 再試行: 同じ判断をやり直した回数とその理由
```

## Reflection プロンプトテンプレート

```
あなたは指示文の改善専門家です。以下の情報を分析し、改善した新候補を生成してください。

## 分析対象の候補（C-XXX）
[候補の指示文全文]

## 実行レポート（シナリオ別）
[Subagent から返ってきたレポートをそのまま貼る]

## 分析タスク
1. レポートの「詰まった箇所」「補完した箇所」を読み、根本的な曖昧点を特定する
2. 最も影響の大きい弱点を 1 つ選ぶ（1イテレーション1テーマ）
3. その弱点を修正した新候補 C-YYY の指示文全文を生成する

## 返答形式
- 特定した根本的な曖昧点: [1行]
- 修正テーマ: [1行]
- 修正がどの要件項目に効くか: [項目名を列挙]
- 新候補 C-YYY の全文: [指示文全文]
```

## イテレーション記録フォーマット（マルチ候補表）

各イテレーションをこの形式で記録・報告する:

```
## Iteration N

### 候補別評価表
| 候補ID | 由来 | シナリオA 精度 | シナリオB 精度 | steps | duration | フロンティア |
|---|---|---|---|---|---|---|
| C-000 | 初期 | 80% | 60% | 5 | 22s | 除外（C-001に支配） |
| C-001 | C-000 を変異 | 90% | 65% | 4 | 18s | ✓ |
| C-002 | C-000 を変異 | 75% | 88% | 6 | 25s | ✓ |
| C-003 | C-001×C-002 Merge | 88% | 85% | 5 | 21s | ✓ |

### 今回新出の不明瞭点（候補別）
- C-001 / シナリオA: （なし）
- C-002 / シナリオB: [critical]項目Nが× — [落ちた理由]

### 今回の変異・Merge
- C-001 → C-004 を生成: [Reflection が特定した弱点と修正テーマ]
- C-001 × C-002 → C-003 Merge: [統合した強みの説明]

### 次イテレーションのフロンティア
- C-003, C-004（を次の親候補として並列実行）

（収束判定: 連続X回クリア / あとY回で停止）
```

## 環境制約

新規 subagent を dispatch できない環境（既に subagent として動作中、Task tool が無効など）では**このスキルを適用しない**。

- 代替案 1: ユーザーに別の Claude Code セッションを起動して依頼してもらう
- 代替案 2: 「evaluation skipped: dispatch unavailable」とユーザーに明示報告する
- **NG**: 自己再読で代替する（バイアスが入るため意味がない）

## よくある間違い

| やりがちな間違い | 実態 |
|---|---|
| **「1つのベスト候補だけ維持・上書きすれば良い（Greedy探索）」** | シナリオAに強い候補とシナリオBに強い候補は別物。Greedy探索は局所最適解に陥りやすく、有望な候補を早期に捨てる。必ず候補プールを維持する |
| 「自分で読み直せば同じ」 | 自分が書いた文章の客観視は不可能。必ず新規 subagent を使う |
| 「シナリオは 1 つで充分」 | 1シナリオでは過適合する。最低 2 つ、できれば 3 つ |
| 「不明瞭点ゼロが 1 回出たから終わり」 | 偶然のこともある。連続 2 回で確定 |
| 「Merge は常に良い結果を出す」 | Merge 候補が両親を支配しないことも多い。フロンティアを通じて評価してから採用する |
| 「数値が良いから質的フィードバックは無視」 | 数値だけ追うと指示が削られすぎる。質的フィードバックを主にする |
| 「前回の subagent を再利用しよう」 | 前回の改善を学習している。毎回新規に dispatch する |
| 「もう書き直した方が早い」 | 3回以上フロンティアが縮小しないなら正解。それ以前は設計への逃げ |
