# スキル作成ワークフロー

新しいAgent Skillを作成し、チームで使用可能な状態にする。

## 進捗チェックリスト

- [ ] Step 0: スキルの作成場所を確認
- [ ] Step 1: インテント整理・要件インタビュー
- [ ] Step 2: SKILL.md の作成
- [ ] Step 3: 品質セルフレビュー
- [ ] Step 4: tuning-prompts の実行確認（AskUserQuestion）・完了通知

---

## Step 0: スキルの作成場所を確認

作成場所は `.agents/skills/` 固定とする。

```bash
REPO_ROOT=$(git rev-parse --show-toplevel)
# ベースディレクトリ: $REPO_ROOT/.agents/skills/
# フルパスは Step 1 でスキル名が確定した後に決定する
```

---

## Step 1: インテント整理・要件インタビュー

会話履歴にすでに情報がある場合は抽出してから確認を取る。不明点は `AskUserQuestion` を使って選択式で確認することを検討する。

**確認する項目:**
1. **目的**: このスキルで何を実現したいか。どんな成果物や動作が期待されるか
2. **トリガー条件**: どんなユーザーの発言や状況でこのスキルが起動すべきか
3. **スキル名**: 動名詞形（例: `reviewing-code`, `generating-reports`）で決める。会話の内容からスキル名を推定し、**候補を提案してユーザーに確認する**。複数の候補が考えられる場合は `AskUserQuestion` で選択式にする
4. **出力形式・制約**: ファイル出力があるか、特定のフォーマットがあるか
5. **依存関係**: 使用するツール・MCP・スクリプトがあるか
6. **引数の有無**: このスキルがユーザーから引数を受け取る必要があるか

### 引数がある場合に追加で確認する項目

引数が必要そうなら、**この段階で意味レベルの要件だけを確定する。YAMLの形はまだ書かない。**

最低限、各引数について以下を確認する:

- 引数名（ユーザーに見せる名前）
- 何を表すか
- 必須か任意か
- 1つだけか、複数候補を取りうるか
- 例（1つでよい）

例:
- `target-repo`: 調査対象のリポジトリ。必須。例: `android`
- `days`: 過去何日分を見るか。任意。例: `30`

### 引数に関する重要ルール

- **Step 1 では引数の意味だけを決める。`arguments` フィールドのYAML構造を推測で設計しない。**
- 引数が不要なら、Step 2 で `argument-hint` と `arguments` は追加しない。
- 引数が必要でも、正式なスキーマが参照先で確認できない限り、**推測で `arguments` を書かない**。
- 参照先を確認しても `arguments` の正式な書式が不明な場合は、`argument-hint` のみ使う案を優先し、その可否をユーザーに確認する。

**Step 1 末尾: フルパスの確定**

スキル名が確定したら、フルパスを確定し、以降のステップで使用する:

```bash
SKILL_DIR=$REPO_ROOT/.agents/skills/{skill-name}/
# 例: $REPO_ROOT/.agents/skills/reviewing-code/
```

## Step 2: SKILL.md の作成

### ディレクトリ構造

```
{skill-name}/
├── SKILL.md          # 必須。500行以下
├── references/       # 詳細情報が多い場合のみ作成
│   └── detail.md     # 100行超の場合は目次を付ける
├── examples/         # 期待する出力例がある場合のみ作成
│   └── sample.md     # 期待する出力の例
└── scripts/          # スキルが使うスクリプトがある場合のみ作成
    └── run-something.sh  # シェルスクリプト（.sh）または JavaScript（.js）
```

**scripts/ のルール:**
- スクリプトが必要な場合、会話履歴にすでに種別の情報がなければ `AskUserQuestion` でシェルスクリプト（`.sh`）か JavaScript（`.js`）かを確認する
  - `.sh` 推奨: CLIツール呼び出し・単純な自動化
  - `.js` 推奨: 複雑なロジック・Node.js エコシステムを使う処理
- SKILL.md 内にスクリプトのパスと役割を明記する

### フロントマターのルール

```yaml
---
name: {skill-name}
description: {説明文}
---
```

利用可能なフィールド・`argument-hint`/`arguments` の決定ルール・`model`/`description`/`name` のルールは [`references/frontmatter-fields.md`](references/frontmatter-fields.md) を参照する（**引数あり**の場合は必ず読む）。

フィールド名の早見表:

| フィールド | 必須 |
|---|---|
| `name` | Required |
| `description` | Required |
| `when_to_use` | Optional |
| `argument-hint` | Optional |
| `arguments` | Optional |
| `disable-model-invocation` | Optional |
| `user-invocable` | Optional |
| `allowed-tools` | Optional |
| `model` | Optional |
| `effort` | Optional |
| `context` | Optional |
| `agent` | Optional |
| `hooks` | Optional |
| `paths` | Optional |
| `shell` | Optional |

### ボディの書き方

**簡潔さを最優先する。** エージェントがすでに知っていることは書かない。各文・各セクションに「これはなぜ必要か？」と問い直す。

```markdown
# スキルのタイトル

一言で何をするスキルか説明する（省略可）。

## 進捗チェックリスト（複数ステップがある場合）
- [ ] ステップ1: ...
- [ ] ステップ2: ...

## ステップ1: ...

具体的な指示。WHY を説明し、過度な MUST/NEVER を避ける。
```

**良い例:** `pdfplumberでテキストを抽出する：` + コードスニペット
**悪い例:** `PDF（ポータブルドキュメントフォーマット）とは一般的なファイル形式で...`

**参照ファイルのルール:**
- SKILL.md → references/ の1レベル深い参照のみ（それ以上のネストはNG）
- 100行を超える参照ファイルには目次を付ける
- 「いつ読むか」のガイダンスをSKILL.md内に書く

## Step 3: 品質セルフレビュー

SKILL.md を書いたら、以下のチェックリストで品質を確認する。
詳細な評価基準は `reviewing-skills` スキルに準拠する。

### コア品質

- [ ] `name` フィールドが具体的で**動名詞形**（動詞 + -ing）を使用している
- [ ] `name` が曖昧・過度に一般的でない（`helper`、`utils` 等でない）
- [ ] `description` がスキルが何をするかを具体的に説明している
- [ ] `description` がいつ使用するかを具体的に説明している
- [ ] `description` が三人称で書かれている
- [ ] `description` に具体的なキーワードが含まれている
- [ ] 引数がない場合 `argument-hint` / `arguments` を追加していない
- [ ] 引数がある場合、`argument-hint` フィールドが存在し値がクォートで囲まれている
- [ ] 引数がある場合、本文にも各引数の意味・必須/任意・例が書かれている
- [ ] `arguments` を使った場合、構造が `references/frontmatter-fields.md` の記法と一致している
- [ ] `arguments` に参照先にないキーや型を推測で追加していない
- [ ] `paths` を使用した場合、glob パターンがプロジェクト構造（単一 vs マルチモジュール）に合っている
- [ ] `model` を使用した場合、値がユーザー確認済みの実在するモデルスラッグである
- [ ] SKILL.mdボディが500行以下
- [ ] 追加の詳細情報は別ファイルに分離されている（必要な場合）
- [ ] 時間に敏感な情報がない
- [ ] スキル全体で一貫した用語を使用している
- [ ] 例が抽象的でなく具体的
- [ ] ファイル参照はSKILL.mdから1レベル深い
- [ ] 段階的開示が適切に使用されている
- [ ] 複雑なワークフローに明確なステップがある
- [ ] Windowsスタイルのパス（`\`）が使われていない

### コードとスクリプト（該当する場合）

- [ ] スクリプトが問題を解決し、AIに任せていない
- [ ] エラー処理が明示的で有用なメッセージを含む
- [ ] 必要なパッケージがリストされ、インストール方法も明示されている
- [ ] スクリプトに明確なドキュメントがある
- [ ] スクリプトの実行意図が明確

問題が見つかった場合はSKILL.mdを修正してから次に進む。

---

## Step 4: tuning-prompts の実行確認・完了通知

SKILL.md が完成したら、`AskUserQuestion` ツールで以下を確認する:

```
質問: 「tuning-prompts を実行しますか？」

説明: tuning-prompts はパレート探索型ワークフローでSKILL.mdを反復改善します。
⚠️ 注意: 複数のSubagentを並列実行するためトークンを大量消費します。
新規作成・大幅改訂直後のスキルに特に効果的ですが、コストが高い処理です。

選択肢:
- はい（実行する）
- いいえ（スキップする）
```

「はい」の場合は `tuning-prompts` スキルを起動する。
tuning-prompts は内部で再度ユーザー確認を行うため、二重確認となるが問題ない。

完了後、ユーザーに伝える:

> ✅ スキルの作成が完了しました。
>
> **スキルを使用するには AI Agent のセッションを再起動してください。**
> 再起動しないと新しいスキルがロードされません。
