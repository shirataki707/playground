# コミット

変更を論理単位に分けてコミットし、必要に応じて AGENTS.md・README・ADR を更新する。

## 進捗チェックリスト

- [ ] Step 1: 差分の把握と粒度設計
- [ ] Step 2: 論理単位ごとにコミット
- [ ] Step 3: AGENTS.md・README の更新判断
- [ ] Step 4: ADR の更新判断

---

## Step 1: 差分の把握と粒度設計

```bash
git status
git diff
git diff --cached
git log --oneline -5
```

変更内容を以下の観点で分類する:

- **機能単位**: 新モジュール・新機能・バグ修正はそれぞれ独立したコミット
- **ビルド設定**: Convention Plugin・Gradle 設定変更はまとめる
- **ツール設定**: `.agents/`・`.claude/` 等の設定変更は別コミット
- **IDE 設定**: `.idea/` は関連する機能コミットに含める（または別コミット）

まだステージされていないファイルがある場合は `git add <files>` で適切に選択してからコミットする（`git add .` は全ファイルを追加するため使用に注意）。

---

## Step 2: 論理単位ごとにコミット

各コミットは以下の形式で作成する:

```bash
git add <関連ファイル>
git commit -m "$(cat <<'EOF'
<type>: <概要（英語・命令形）>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

コミットメッセージの type:
- `feat` / `add`: 新機能・新モジュール追加
- `fix`: バグ修正
- `refactor`: 動作変更を伴わない構造変更
- `docs`: ドキュメントのみの変更
- `build`: ビルド設定・Convention Plugin
- `skill`: `.agents/skills/` の変更

---

## Step 3: AGENTS.md・README の更新判断

Step 2 でコミットした変更を振り返り、開発者向けドキュメントが陳腐化していないか確認する。

### AGENTS.md を更新すべきケース

| 変化の種類 | 更新箇所の目安 |
|---|---|
| モジュールを追加・削除・リネームした | モジュール構成セクション |
| Convention Plugin を追加・変更した | Convention Plugins テーブル |
| ビルド・リント手順が変わった | ビルドとリントセクション |
| コード規約・制約が変わった | コード規約セクション |

### README を更新すべきケース

プロジェクトに `README.md` が存在する場合、以下の変化で更新する:

- セットアップ手順・前提条件が変わった
- 主要な機能・画面が追加・削除された
- 外部から見えるモジュール構成や依存関係が変わった

### スキップしてよいケース

- バグ修正・小規模リファクタで外部から見える構造に変化なし
- 既存パターンに沿った機能追加（ドキュメントに記載済みの手順の繰り返し）
- テストのみの変更

### 操作

変更があれば直接編集し、Step 2 と同じコミットに含める **か**、`docs: update AGENTS.md` として別コミットにする。

---

## Step 4: ADR の更新判断

Step 2 でコミットした変更を振り返り、アーキテクチャ上の決定が含まれるか判断する。

### 更新が必要なケース

| 変化の種類 | 対応 |
|---|---|
| 新しいライブラリ・技術を採用した | 新規 ADR を作成する |
| モジュール構成・パッケージ構造を変更した | 新規 ADR を作成 **または** 既存 ADR を Superseded にする |
| 既存の決定を覆した・方針を変更した | 既存 ADR を Superseded にして新規 ADR を作成する |
| 設計パターンを新たに確立した | 新規 ADR を作成する |

### スキップしてよいケース

- バグ修正・小規模リファクタ（アーキテクチャ決定ではない）
- 既存パターンに沿った機能追加（新しい決定がない）
- テスト・ドキュメントのみの変更
- `.idea/` 等のツール設定変更

### 判断に迷う場合

コミット内容から判断できない場合は `AskUserQuestion` でユーザーに確認する。

### ADR 操作

`docs/adr/` 内の既存 ADR を確認してから操作する:

```bash
ls docs/adr/
```

- 新規作成・更新の手順は `managing-adrs` スキルの手順に従う
- ADR ファイルの変更後は Step 3 の変更と合わせて `docs:` コミットにまとめるか、単独で `docs: update ADR` コミットを作成する
