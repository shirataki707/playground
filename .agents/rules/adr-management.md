# ADR (Architecture Decision Records) の自動管理

実装が完了したタイミングで、**その変更が ADR として記録すべき技術的決定かどうかを自分で判断**し、必要であれば `managing-adrs` スキルを起動する。

## 判断のタイミング

コミット直前または実装完了後に以下を自問する:

> 「この変更は、将来の開発者や AI エージェントが背景を知らなければ理解できない技術的決定を含んでいるか？」

## ADR を作成すべき変更（create）

- 新しい主要ライブラリや Jetpack コンポーネントを `build.gradle.kts` / `libs.versions.toml` に追加した
- データ層・UI 層・ナビゲーション層のアーキテクチャ方針を新たに定めた
- 複数の選択肢を比較検討した結果、特定の技術・設計を採用した

## ADR を更新すべき変更（update）

- 過去に採用した技術・ライブラリを別のものに置き換えた
- 既存のアーキテクチャ方針を覆す実装を行った

## ADR 不要な変更（スキップ）

- 既存パターンに沿った feature モジュール追加・バグ修正・UI の微調整
- パッチ・マイナーバージョンの依存関係更新
- リファクタリング（外部から見た設計方針の変更を伴わないもの）

## 判断結果に応じたアクション

| 判断 | アクション |
|---|---|
| ADR を作成すべき | `managing-adrs` スキルを `create` アクションで起動する |
| ADR を更新すべき | `managing-adrs` スキルを `update` アクションで起動する |
| 不要 | 何もしない（理由を一言だけ添えて次に進む） |
